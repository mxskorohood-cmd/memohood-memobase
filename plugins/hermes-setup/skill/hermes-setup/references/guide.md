# hermes-setup — методичка

Подробное справочное дополнение к `SKILL.md`. Читай сюда, когда нужно понять,
что именно происходит на конкретном шаге мастера, почему ключ отклонён, что
пишется в `config.yaml`/`.env`, или продиагностировать, почему что-то не
работает. Все факты сверены с кодом плагина
(`D:/hermes-fable/plugins/hermes-setup/`).

## 1. Архитектура в двух словах

Мастер — конечный автомат (`wizard.py`), state хранится **только в памяти
процесса** (обычный словарь, ключ — `chat_id`), не в файле: перезапуск Hermes
посреди настройки просто сбрасывает текущий шаг (но не то, что уже сохранено в
`config.yaml`/`.env`) — начать заново нужно командой `/setup`.

Один и тот же набор функций-шагов обслуживает два разных входа:

- **Telegram / другой шлюз** — хук `pre_gateway_dispatch` (`on_gateway_dispatch`)
  перехватывает `/setup` и любой ответ, пока мастер активен для этого чата,
  **до** того, как сообщение увидит модель. Ключ состояния — реальный
  `chat_id` из Telegram.
- **CLI** — слэш-команда `/setup [ответ]` (`cli_setup_command`). У обработчика
  команд в hermes нет понятия «чат» вообще — только текст аргументов, поэтому
  CLI-путь использует фиксированный псевдо-`chat_id` (`"cli"`). Коллизий с
  настоящими (числовыми) `chat_id` из Telegram быть не может — в CLI-сеансе
  всегда ровно один локальный оператор.

Оба пути НЕ используют блокирующий `input()` — даже в реальном терминале.
Причина: обработчик слэш-команды по контракту плагинов hermes может быть
вызван и внутри процесса шлюза тоже (если `pre_gateway_dispatch` по какой-то
причине не перехватит сообщение первым) — там нет интерактивного `stdin`, и
блокирующий `input()` подвесил бы поток шлюза навсегда. Поэтому CLI тоже
устроен как «один вызов — один ответ»: `/setup <ответ>`.

## 2. Шаги мастера от начала до конца

1. **`/setup`** → приветствие + меню: «1 — всё по порядку» / «2 — выбрать
   один плагин».
2. **Выбор 1** → очередь `["token-guard", "memobase", "memohood"]` (это и есть
   порядок `registry.PLUGIN_ORDER`). **Выбор 2** → отдельный вопрос «какой
   плагин?» с нумерованным списком, дальше очередь из одного элемента.
3. **Для каждого плагина в очереди** (`_enter_plugin`):
   - плагин включается в конфиге (`plugins.enabled` для token-guard/MemoBase,
     `memory.provider: memohood` для MemoHood — см. раздел 4);
   - если папки плагина ещё нет на диске (`registry.discover_plugin_dirs`) —
     к описанию плагина добавляется одна строка-предупреждение, но настройка
     всё равно продолжается;
   - считаются недостающие ключи (`envfile.scan_keys` по `.env`) — уже
     сохранённые ключи повторно не спрашиваются;
   - если недостающих ключей нет (token-guard: их 0 вообще) — шаг **рекурсивно
     сворачивается** в следующий плагин очереди, без «пустого» вопроса;
   - если есть — задаётся вопрос по первому недостающему ключу (метафора +
     формат + «пришлите значение… или напишите «пропустить»»).
4. **На каждый присланный ключ** (`_advance_key`):
   - «пропустить» → ключ не сохраняется, следующий вопрос;
   - `registry.validate_key` проверяет формат (и «не похож ли на ключ от
     другого сервиса») — при провале мастер объясняет, что не так, и
     переспрашивает тот же ключ;
   - при успехе — `envfile.upsert_env_value` пишет в `.env`, затем
     `registry.live_check_key` делает (если для ключа есть проверка) один
     живой HTTP-запрос и честно говорит, сработал ключ или нет;
   - в любом случае — совет удалить сообщение с ключом из истории чата;
   - когда ключи плагина закончились — переход к следующему плагину очереди.
5. **Очередь исчерпана** → `_DONE_MESSAGE`: «Готово!.. перезапустите Hermes» —
   и состояние мастера для этого чата стирается (`clear_wizard`).
6. **В любой момент**: слово из `{отмена, стоп, cancel, stop, выход, exit}`
   (без учёта регистра, без хвостовых `.,!`) → мастер останавливается сразу,
   ничего больше не дописывая, состояние стирается. Это **не** то же самое,
   что шаг 5 — отмена не просит перезапустить Hermes, потому что ничего
   финального не произошло.
7. **15 минут без ответа** (`_WIZARD_TIMEOUT_SECONDS = 15 * 60`) → при
   следующем сообщении мастер сам обнаруживает просрочку, сбрасывает
   состояние и просит начать заново командой `/setup`. Тоже не «готово» —
   восстановить прогресс можно, повторно ответив на уже настроенные шаги
   ничего не придётся: то, что уже сохранено в `config.yaml`/`.env`, никуда
   не делось.

**Важная деталь безопасности:** мастер привязан не только к `chat_id`, но и к
`user_id`, который его запустил (`entry["user_id"]`). Если в том же чате
ответит кто-то другой — `handle_message` вернёт `None` (сообщение пройдёт как
обычное, мастер его не перехватит). В групповом чате это значит: чужой
человек не может ни перехватить, ни «помочь» ответить за того, кто запустил
`/setup`.

## 3. Ключи — таблица

Все специфичные знания о ключах живут в `registry.KEY_SPECS`. Столбец «живая
проверка» — это одноразовый HTTP-запрос сразу после сохранения (таймаут 15
секунд, без повторов); столбец «детектор чужого ключа» — участвует ли формат
ключа в проверке «это не то, что вы хотели прислать» (`registry.classify`).

| Ключ | Для чего (метафора мастера) | Формат | Живая проверка | Детектор чужого ключа |
|---|---|---|---|---|
| `CLOUDFLARE_ACCOUNT_ID` | «Переводчик текста в числа-смыслы» — ID аккаунта Cloudflare | 32 hex-символа (`[0-9a-fA-F]{32}`) | нет | да |
| `CLOUDFLARE_API_TOKEN` | Второй ключ от того же Cloudflare — токен API | непустая строка без пробелов | да (`/user/tokens/verify`) | нет — формат неотличим от произвольного токена |
| `COHERE_API_KEY` | «Строгий редактор» — отбирает самое нужное наверх | непустая строка без пробелов | да (`GET /v1/models`) | нет |
| `GEMINI_API_KEY` | «Младший редактор» — разруливает спорные факты | начинается с `AIza`, ≥30 символов, без пробелов | да (`GET /v1beta/models`) | да |
| `SCRAPECREATORS_API_KEY` | Достаёт субтитры и списки видео с YouTube | непустая строка без пробелов | нет | нет |
| `APIFY_TOKEN` | Запасной путь — скачивает аудио, если субтитров нет | начинается с `apify_`, ≥10 символов, без пробелов | да (`GET /v2/users/me`) | да |
| `GROQ_API_KEY` | Быстро расшифровывает аудио в текст (Whisper) | начинается с `gsk_`, ≥10 символов, без пробелов | да (`GET /openai/v1/models`) | да |

«Детектор чужого ключа» специально не участвует для ключей с общим форматом
(`COHERE_API_KEY`, `CLOUDFLARE_API_TOKEN`, `SCRAPECREATORS_API_KEY`) — их
формат неотличим от «любой случайной строки», поэтому они никогда не могут
быть целью межключевого сравнения, только источником (если такой ключ
случайно попал туда, где ждали, например, Gemini — вот это как раз поймает
детектор gemini-ключа).

Каждый живой HTTP-запрос отправляется с реалистичным браузерным
`User-Agent` — некоторые API (в частности Cloudflare) отклоняют запросы без
него.

## 4. Что именно меняется в конфиге по каждому плагину

`registry.PLUGINS` — реестр из трёх плагинов, `registry.PLUGIN_ORDER` задаёт
порядок по умолчанию: `token-guard → MemoBase → MemoHood`.

- **token-guard** — 0 ключей. Единственное действие — добавить `token-guard`
  в список `plugins.enabled` (если его там ещё нет). Раз ключей нет, шаг
  сразу сворачивается в следующий плагин очереди.
- **MemoBase** (slug `memobase`) — 7 ключей (см. таблицу выше). Действие — тоже просто
  добавление `memobase` в `plugins.enabled`.
- **MemoHood** (slug `memohood`) — 4 ключа (`CLOUDFLARE_ACCOUNT_ID`, `CLOUDFLARE_API_TOKEN`,
  `COHERE_API_KEY`, `GEMINI_API_KEY` — все четыре уже входят в список
  MemoBase, поэтому если MemoBase настраивали раньше, повторно их спрашивать
  не будут). Действие отличается от двух других: MemoHood — **провайдер памяти**,
  поэтому вместо `plugins.enabled` мастер выставляет `memory.provider: memohood` и
  дописывает блок `memory.memohood.*` — дефолтные значения из `MEMOHOOD_DEFAULT_CONFIG`
  (`wizard.py`), но **существующие значения пользователя побеждают** дефолты
  (рекурсивный `_deep_merge`, где база — дефолт, а поверх накладывается уже
  существующий `memory.memohood`, если он был).

И `plugins.enabled`, и `memory.provider`/`memory.memohood.*` пишутся только через
официальный `hermes_cli.config.load_config`/`save_config` — никогда руками,
никогда мимо стандартного механизма. Если `save_config` бросает исключение
(в том числе `SystemExit` — это отдельно отловлено, потому что
`SystemExit` — это `BaseException`, а не `Exception`, и обычный
`except Exception` его не поймает; типичная причина — ключ конфига
заблокирован managed-конфигурацией, например корпоративная/NixOS-сборка) —
мастер это ловит, **не падает**, добавляет к описанию плагина одну строку
«не удалось включить автоматически — включите вручную позже», и всё равно
идёт дальше спрашивать и сохранять ключи API.

## 5. Почему это безопасно (маскирование и `.env`)

- **Маска** (`envfile.mask_key`) — первые 4 символа значения + «…»
  (`AIza…`); пустая строка на входе даёт пустую строку. Полный ключ никогда
  не возвращается в вызывающий код мастера для показа в чате.
- **`.env`-апсертер** (`envfile.upsert_env_value`) обрабатывает три случая по
  приоритету: (1) уже есть активная (не закомментированная) строка
  `KEY=...` — значение заменяется на месте; (2) активной строки нет, но есть
  закомментированный плейсхолдер (`# KEY=` / `#KEY=`) — он раскомментируется
  и заполняется, сохраняя исходную позицию в файле; (3) ни того, ни другого —
  новая строка дописывается в конец файла. Всегда UTF-8; файл и родительская
  папка создаются, если их не было.
- Значения ключей **никогда** не попадают в логи и текст исключений модулей
  `envfile.py`/`registry.py` — это явное правило, соблюдённое по всему коду
  (сообщения об ошибках содержат текст исключения HTTP-библиотеки, а не сам
  ключ).

## 6. CLI vs Telegram — отличия на практике

- В Telegram после `/setup` можно просто отвечать обычным текстом — мастер
  сам понимает, что это ответ ему, пока не придёт слово отмены или не
  истекут 15 минут.
- В CLI каждый следующий ответ — это аргумент команды: `/setup 1`,
  `/setup AIza...`, `/setup отмена`. `/setup` без аргументов на активном
  мастере **повторно показывает последний заданный вопрос**
  (`_reshow_current_question`), а не продвигает шаг — это осознанно, чтобы
  не потерять место в CLI-сеансе, если пользователь набрал команду без
  ответа по ошибке.
- Оба пути делят один и тот же набор функций-шагов в `wizard.py` — никакой
  отдельной «CLI-версии» логики не существует, поэтому советы из этой
  методички (форматы ключей, поведение отмены/таймаута/пропуска) одинаково
  верны для обоих.

## 7. Диагностика

**«Мастер не понимает мой ответ на меню / выбор плагина»** — на шаге меню
разбирается только **первый символ** ответа («1…» или «2…» — остальное
игнорируется); на шаге выбора плагина, наоборот, нужно **точное** совпадение
(после `strip()` и приведения к нижнему регистру) с номером в списке, slug'ом
или названием плагина целиком. В обоих случаях нераспознанный ответ просто
повторно показывает тот же вопрос — это ожидаемо, не баг.

**«Прислал ключ, а он спрашивает тот же самый»** — `registry.validate_key`
не прошёл: либо формат не совпал (см. таблицу ключей выше), либо значение
похоже на ключ ДРУГОГО сервиса (сработал детектор `classify`) — в обоих
случаях мастер объясняет причину текстом прямо в ответе.

**«Написал «пропустить», а он всё равно спрашивает тот же ключ»** — слово
должно совпасть точно (после `strip()` + нижний регистр) со словом
«пропустить» — лишние слова вокруг («давай пропустить», «пропустить его»)
не распознаются как команда пропуска.

**«Отменил, а конфиг всё равно изменился»** — ожидаемо: отмена останавливает
только ОСТАВШИЕСЯ вопросы. Всё, что было сохранено в `config.yaml`/`.env` на
предыдущих, уже отвеченных шагах ДО отмены, не откатывается — мастер не
хранит транзакцию по всей сессии, каждый шаг применяется сразу.

**«После `/setup` плагин всё равно не работает»** — самая частая причина:
не был перезапущен Hermes. Мастер сам не умеет перезапустить процесс — это
единственный шаг, который остаётся за пользователем.

**«Живая проверка ключа не прошла, хотя ключ верный»** — проверка
одноразовая, без повторов, с таймаутом 15 секунд
(`registry.LIVE_CHECK_TIMEOUT_SECONDS`); при сетевой ошибке или таймауте
мастер честно пишет «не удалось проверить» — но **ключ всё равно
сохраняется** (проверка — это диагностика, а не условие сохранения). «Не
удалось проверить» ≠ «ключ неверный».

**«Мастер завис / долго не отвечал»** — если прошло больше 15 минут с
последнего сообщения, при следующем ответе мастер сам обнаружит просрочку,
сбросит состояние и попросит начать заново `/setup`. Всё, что уже было
сохранено в `config.yaml`/`.env`, остаётся на месте — начинать с нуля не
придётся.

**«Включение плагина в конфиге не произошло, хотя ключи сохранились»** —
значит `_enable_plugin_config` поймал исключение (см. раздел 4) — обычно
managed-конфигурация с заблокированным ключом. Ключи API при этом всё равно
в `.env`; `plugins.enabled`/`memory.provider` придётся дописать вручную,
администратору окружения.

## 8. Чего мастер не делает (сознательно)

- **Не копирует файлы плагинов.** hermes-setup настраивает только конфиг
  (`config.yaml` + `.env`) — папки token-guard/MemoBase/MemoHood должны быть на
  диске до перезапуска Hermes, иначе включённая запись в конфиге ничего не
  подхватит. `discover_plugin_dirs` — это подсказка (одна строка
  предупреждения), а не жёсткая проверка: настроить и сохранить ключи можно
  и заранее.
- **Не перезапускает Hermes сам.** Финальное сообщение просит сделать это
  вручную — у плагина нет доступа к процессу хоста извне.
- **Не хранит прогресс на диске.** Состояние — только в памяти процесса;
  перезапуск Hermes посреди `/setup` теряет текущий шаг (но не то, что уже
  сохранено в конфиге/`.env`).
- **Не откатывает уже применённые шаги при отмене.** Отмена останавливает
  только то, что ЕЩЁ не было сделано — см. раздел 7.
- **Не имеет собственных API-ключей.** `requires_env: []` в `plugin.yaml` —
  hermes-setup только ЗАПИСЫВАЕТ чужие ключи в `.env`, никогда не читает их
  для себя.

## 9. Как это соотносится с онбордингом других плагинов

MemoBase и MemoHood умеют настраиваться и без hermes-setup, но по-разному:

- **MemoBase** — свой мастер запускается двумя способами с ОДНИМ и тем же
  общим ядром, чтобы они не расходились: `/memobase setup` в Telegram или
  `hermes memobase setup` в терминале. Это отдельный путь, не завязанный на
  hermes-setup.
- **MemoHood** — терминальный путь `hermes memohood setup` тоже существует и работает
  независимо. Но у MemoHood нет и не может быть аналога `/memobase setup` в чате: MemoHood —
  провайдер памяти, и архитектурно не имеет доступа к чату, пока
  `memory.provider` уже не указывает на неё. Именно этот разрыв закрывает
  hermes-setup — он общий, включается вручную один раз и умеет переключить
  `memory.provider` на memohood за пользователя, после чего MemoHood уже может
  участвовать в диалоге.

Вывод для практики: если пользователь уже настраивал ключи через `/memobase setup`
или `hermes memohood setup`, hermes-setup их не спросит повторно — `.env`
проверяется по факту наличия значения, а не по тому, кто его туда записал.
