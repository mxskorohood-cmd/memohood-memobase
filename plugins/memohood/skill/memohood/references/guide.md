# MemoHood — методичка

Подробное справочное дополнение к `SKILL.md`. Читай сюда, когда нужно понять
смысл конкретного поля вывода, механику supersede/pinned/decay или
продиагностировать, почему память ведёт себя не так, как ожидалось. Все
факты сверены с кодом плагина (`D:/hermes-fable/plugins/memohood/`) и
`DESIGN_v1.md`.

## 1. Архитектура в двух словах

- **Перед каждым ответом** — `prefetch`: гибридный поиск (FTS5 с русским
  стеммингом + вектор BGE-M3 через Cloudflare + RRF, опционально Cohere
  rerank) находит релевантные записи и историю, результат вставляется в
  промпт отдельным блоком `<memory-context>`. Это никогда не блокирует и не
  переписывает уже сохранённую историю сообщений.
- **После каждого ответа** — `sync_turn`: работает в фоновом потоке (не
  задерживает ответ пользователю), запускает двухступенчатую экстракцию
  фактов (`capture.py`) и решает — добавить запись, заменить старую
  (supersede) или ничего не делать (дубль/шум).
- **Ночью, по расписанию `hermes cron`** — консолидация (`consolidate.py`):
  угасание confidence по кривой Эббингауза (кроме pinned), дедуп похожих
  записей, свёртка старых в сводки день→неделя→месяц, перестройка
  полнотекстового индекса.

## 2. Инструменты — расшифровка параметров и вывода

### `memohood_search(query, k=8)`

Сырой поиск без форматирования под обычный recall-блок. Возвращает список
найденных записей построчно: `[capture:{id}] kind=... score=... source=...`.
Полезен, когда нужен именно `id` записи для последующего `memohood_fetch`, а не
готовый текст для ответа пользователю.

### `memohood_fetch(capture_id)`

Одна запись по id: содержимое, `kind`, `notability`, `source` (откуда взят
факт — извлечён явно или выведен), `pinned`, флаг «замещена ли» и поле
`history` — накопленный текст прошлых версий с датами, если запись хоть раз
проходила через supersede.

### `memohood_recall(query, k=10)`

То же самое, что автоматический `prefetch` перед каждым ходом, но по
явному запросу и с настраиваемым `k`. Используй, когда нужно вспомнить
что-то по конкретной теме здесь и сейчас, а не полагаться на то, что
авто-recall уже подмешал нужное в контекст.

### `memohood_stats()`

Без параметров. Формат вывода:

```
Активных записей: {N} (закреплено: {N})
  {kind}: {N}          -- по каждому kind, по убыванию count
Watermark индексации истории (messages_fts): {value}
Расход за 30 дней ({provider}): ${spent:.4f} из ${ceiling:.2f}    -- по каждому провайдеру
```

«Активных» — записи, у которых `invalidated_at IS NULL` (не заменены и не
заархивированы). Watermark — до какого сообщения истории уже построен
полнотекстовый индекс (нужен для recall по старым сообщениям).

### `memohood_capture(content, kind, pinned)`

`kind` — один из: `persona`, `event`, `preference`, `decision`,
`correction`, `fact`, `instruction`. `pinned` — булево. Сохраняет факт в
обход авто-извлечения (без вызова Gemini на классификацию) — используй,
когда пользователь прямо просит что-то запомнить и нужна гарантия, а не
надежда на то, что сигнальный скоринг поймает реплику сам.

Даже у явного `memohood_capture` при сохранении отрабатывает supersede-проверка
против уже существующих похожих записей (см. раздел 5) — дубликат не
создаст вторую копию того же факта.

### `recall_all(query, k=6)`

Ищет одновременно по captures памяти (через тот же гибридный поиск) и по
коллекциям MemoBase — если плагин MemoBase установлен и загружен
(проверяется best-effort, без жёсткой зависимости; если не установлен —
явно сообщает об этом и ищет только по памяти). Каждый результат помечен
источником. Если среди найденных записей памяти есть созданная не более
14 дней назад — добавляется предупреждение, что при конфликте между
памятью и базой знаний стоит доверять более свежей памяти.

## 3. CLI-команды — примеры вывода

### `hermes memohood status` / `hermes memohood stats`

Алиасы. Формат вывода — см. `memohood_stats()` выше.

### `hermes memohood reindex`

```
Переиндексация captures начата (embed_signature: cloudflare:@cf/baai/bge-m3:1024)...
Переиндексация завершена: 47 записей переэмбеддено, векторный индекс готов: True.
```

Нужен после смены `memory.memohood.embedder.model` — старые векторы, посчитанные
прежней моделью, несравнимы с новыми запросами.

### `hermes memohood seed [--dry-run]`

```
Индекс истории (messages_fts): watermark 0 -> 1842, проиндексировано 1842 сообщений за 4 батч(ей) (пропущено пустых: 12).
Извлечение ФАКТОВ (captures) из старой истории через LLM пока не реализовано отдельным проходом -- capture.process_turn работает только для НОВЫХ ходов. messages_fts уже наполнен и доступен для recall (memohood_recall/recall_all/prefetch).
```

Важно: только индексация для полнотекстового поиска, **не** извлечение
новых captures из старых сообщений. `--dry-run` показывает watermark, ничего
не индексируя — удобно перед реальным запуском на большой истории.

### `hermes memohood consolidate`

```
Запуск ночной консолидации (decay/dedup/rollup/FTS-rebuild)...
decay: обновлено=41, архивировано=3
dedup: объединено=2
rollup: day=5 week=1 month=0
fts_rebuild: перестроено=47
```

Ручной запуск того, что штатно должно идти по расписанию `hermes cron` —
плагин сам никакого cron-триггера не создаёт, это нужно настроить отдельно.

### `hermes memohood setup`

Пошаговый мастер онбординга ключей — не связанный с captures/consolidate
инструмент, а разовая настройка `.env`. Проходит четыре шага, каждый
можно пропустить Enter'ом: Cloudflare (эмбеддинги, с объяснением, зачем
нужен векторный поиск по смыслу), Cohere (реранкер — объясняется как
«строгий редактор», пересортировывающий найденное так, чтобы самое нужное
оказалось сверху), Gemini (LLM для спорных фактов и ночной консолидации) и
проверка python-зависимостей (`sqlite-vec`, `PyStemmer`, `ftfy`,
`requests`). На каждом ключевом шаге можно попросить один живой запрос к
провайдеру, чтобы убедиться, что ключ рабочий, прежде чем сохранять. Ключ
никогда не печатается в консоли целиком — только первые символы с
многоточием (`mask_key`), и в `.env` он тоже уходит в этом виде в итоговом
логе действия (`добавлено`/`заменено`/`раскомментировано`).

Мастер работает только с `HERMES_HOME/.env` — он **не** пишет в
`config.yaml`: ни `memory.provider`, ни секцию `memory.memohood` он не трогает.
Это по-прежнему ручной шаг (см. README.md, «Быстрый старт», пункт 2) или
задача `hermes-setup` — единственного из плагинов, которому разрешено
переключать `memory.provider` за пользователя через Telegram `/setup`
(MemoHood сама не может завести Telegram-мастер, пока не станет активным
провайдером памяти — отсюда и существование `hermes-setup`).

Если пользователь спрашивает, как настроить MemoHood, каких ключей не хватает
или что вообще такое реранкер — направляй именно на эту команду, а не на
перечисление переменных `.env` вручную.

## 4. Настройки `memory.memohood.*`

Полная таблица — в README.md (раздел «Настройки»). Значения, которые чаще
всего нужны при диагностике или тонкой настройке:

- `capture_threshold` (по умолчанию `4.0`) — порог бесплатного сигнального
  скоринга. Понизишь — больше реплик будет уходить на Gemini как «спорные»
  (дороже, но точнее на пограничных случаях); повысишь — больше будет
  однозначно пропускаться как шум без вызова LLM.
- `decay.halflife_days.*` — период полураспада confidence по типу факта:
  событие (`event`) — 7 дней, предпочтение/решение/исправление — 90,
  устойчивый факт/персона/инструкция/сводка — 365. Pinned-записи decay не
  затрагивает вообще, независимо от `kind`.
- `decay.floor` (по умолчанию `0.05`) — ниже этого confidence запись
  архивируется при консолидации (не удаляется физически, просто перестаёт
  всплывать в обычном поиске).
- `monthly_ceiling_usd.{cloudflare,cohere,gemini}` (по умолчанию `5` для
  каждого) — при достижении потолка соответствующий внешний вызов для этого
  провайдера просто пропускается до начала следующего месяца.
- `rerank.enabled` (по умолчанию `true`) — можно выключить реранк совсем,
  не только через отсутствие ключа.

## 5. Как работает захват и supersede (детально)

Шесть шагов `capture.py` на каждый ход диалога:

1. **Бесплатный сигнальный скоринг.** Регулярные выражения на RU+EN
   ключевые слова (исправление/решение/предпочтение/явное «запомни»/URL/
   путь), очки суммируются. Счёт `<= 0` — реплика отбрасывается без
   единого вызова LLM. Счёт `>= capture_threshold` (по умолчанию `4.0`) —
   гарантированный захват, тоже без LLM.
2. **Пограничная зона.** Если счёт между «явный шум» и «явный сигнал» —
   ровно один вызов Gemini flash-lite, который решает: стоит ли запоминать,
   какой `kind`, какая `notability`, откуда факт (`EXTRACTED`/`INFERRED`),
   нужен ли `pinned`.
3. **Санитизация.** Входящий текст реплики и исходящий текст факта
   проверяются на попытки внедрения инструкций и утечку секретов; найденные
   секреты редактируются перед сохранением.
4. **Supersede — трёхуровневая проверка похожести на уже существующие
   записи (по косинусному сходству эмбеддингов):**
   - схожесть **≥ 0.95** — считается тем же самым фактом, дубликат без
     вызова LLM, обновляется только «видели в последний раз»;
   - схожесть **< 0.92** — считается независимым новым фактом, тоже без LLM;
   - между **0.92 и 0.95** — ровно один вызов Gemini-судьи, который решает
     явно: `duplicate` / `supersede` / `independent`.

   При `supersede`: старая запись получает `invalidated_at` (перестаёт
   всплывать в обычном поиске) и не удаляется — текст старой версии с датой
   уходит в поле `history` новой записи, которую всегда можно поднять через
   `memohood_fetch`.
5. **Pinned.** Либо по прямому триггеру в тексте (имя/дата рождения/
   аллергия/диагноз/группа крови/«запомни навсегда» и англоязычные
   аналоги), либо по флагу, который вернул Gemini на шаге 2. Pinned-записи
   полностью исключены из decay на ночной консолидации.
6. **Эмбеддинг и индексация.** Запись эмбеддится (Cloudflare) для
   векторного плеча поиска и попадает в полнотекстовый индекс с русским
   стеммингом. Если эмбеддер недоступен (нет ключей/сети) — запись всё
   равно попадает только в полнотекстовый индекс, просто без векторного
   плеча.

## 6. Типовые сценарии

- **Пользователь спрашивает «что ты обо мне помнишь».** Вызови
  `memohood_recall` (по теме разговора, если она есть) или `memohood_stats()` (если
  вопрос именно про объём/состав памяти). Отвечай строго по реально
  вернувшимся данным — если пусто, честно скажи, что пока нечего вспомнить.
- **Пользователь прямо просит что-то запомнить.** В подавляющем большинстве
  случаев реплика уже содержит явный сигнал-триггер, и авто-захват сработает
  сам после ответа. Если нужна гарантия прямо сейчас (например, пользователь
  хочет явное подтверждение) — вызови `memohood_capture` с подходящим `kind`.
- **Пользователь говорит, что воспоминание неверное или устарело.** Нет
  инструмента удаления — объясни это прямо и предложи назвать правильную
  версию факта («на самом деле...»). Supersede заменит старую запись новой;
  старая версия останется только в `history`, доступной через `memohood_fetch`,
  но перестанет всплывать в обычном recall.
- **Пользователь хочет, чтобы факт не забывался никогда.** Либо реплика уже
  содержит один из pinned-триггеров (имя, дата рождения, аллергия, диагноз,
  «запомни навсегда»), либо вызови `memohood_capture(..., pinned=true)` явно.
- **Нужно поискать сразу и по разговорам, и по документам** (если
  MemoBase тоже установлена) — используй `recall_all`, а не два отдельных
  вызова: он уже метит источник и учитывает приоритет свежести памяти.
- **Пользователь спрашивает про стоимость/объём памяти.** Всегда вызывай
  `memohood_stats()` или `hermes memohood status` — не оценивай на глаз.

## 7. Диагностика

**Пользователь не знает, какие ключи у него настроены, или просит помочь настроить MemoHood с нуля.**
Не перечисляй переменные `.env` вручную — направь на `hermes memohood setup`:
мастер сам пройдёт по всем трём ключам, объяснит каждый простым языком
(включая реранкер), по желанию проверит живым запросом и сохранит в
`.env` в замаскированном виде. Он не трогает `config.yaml` — если ещё не
выставлен `memory.provider: memohood`, это отдельный ручной шаг (или разовая
настройка через `hermes-setup` и `/setup` в Telegram).

**Спорные факты не запоминаются, хотя реплика неоднозначная.**
Проверь `GEMINI_API_KEY` — без него шаг 2 (Gemini на пограничные случаи)
просто пропускается. Явные сигналы (шаг 1) продолжают работать бесплатно.

**Recall находит только точные совпадения слов, не улавливает смысл.**
Проверь `CLOUDFLARE_ACCOUNT_ID`/`CLOUDFLARE_API_TOKEN` — без них векторное
плечо поиска отключается, остаётся только полнотекстовый поиск (FTS5 с
русским стеммингом через PyStemmer — он всё ещё находит разные формы слова,
но не близкие по смыслу разные слова).

**Порядок результатов в топе выглядит немного не тем, что ожидалось.**
Проверь `COHERE_API_KEY` и `memory.memohood.rerank.enabled` — без ключа или при
`enabled: false` финальный реранк пропускается, остаётся чистый порядок RRF.

**`hermes memohood status` показывает расход по провайдеру на пределе потолка.**
Не ошибка — при достижении `monthly_ceiling_usd` для этого провайдера
соответствующий шаг (эмбеддинг/реранк/экстракция) тихо пропускается до
начала следующего месяца. Можно поднять потолок в `config.yaml`.

**Ошибка блокировки базы (`memory.db` locked).**
База — SQLite в режиме WAL с `busy_timeout`, параллельные обращения обычно
просто ждут друг друга. Если блокировка всё же произошла — обычно предыдущий
процесс hermes завершился аварийно. Перезапуск hermes чинит это сам, WAL
восстанавливается при следующем открытии базы.
