# token-guard — методичка

Подробное справочное дополнение к `SKILL.md`. Читай сюда, когда нужно
понять смысл конкретного поля отчёта, правило аудита, безопасно провести
пользователя через включение тумблера или продиагностировать, почему
что-то не работает. Все факты сверены с кодом плагина
(`D:/hermes-fable/plugins/token-guard/`) и `DESIGN.md`.

## 1. Архитектура в двух словах

- **Часть 1 (всегда активна, если плагин включён)** — только наблюдение:
  журнал (`ledger.py`, SQLite), детектор сброса кэша (`cache_guard.py`),
  аудит конфигурации (`audit.py`). По построению не может сделать модель
  «тупее» — она ничего не пишет в конфиг и не меняет поведение агента.
- **Часть 2 (опционально, выключено по умолчанию)** — три тумблера
  (`toggles.py`): `cheap_aux`, `cheap_delegation`, `cache_1h`. Каждый —
  минимальный набор ключей `config.yaml`, включаемый только через
  двухшаговое подтверждение с карточкой риска и откатываемый одной
  командой.

Дольше — про архитектурные причины таких решений (не нужно для
повседневной работы, но полезно, если пользователь спрашивает «почему
именно так»):

- Хуки (`post_api_request`, `api_request_error`, `post_tool_call`)
  вызываются хостом синхронно на каждый запрос/вызов инструмента —
  поэтому запись в журнал всегда один `INSERT`, обёрнутый в
  `try/except`, без исключений наружу.
- Доллары не считаются в хуках. `hermes` сам оценивает стоимость сессии
  (`estimated_cost_usd` в `hermes_state.SessionDB`) — плагин читает её
  в режиме только для чтения (`read_only=True`, без блокировки на
  запись) вместо того, чтобы держать свою таблицу цен, которая могла бы
  разойтись с реальной.
- Тумблеры пишут конфиг через `hermes_cli.config.set_config_value`
  (безопасный «точечный» сеттер по dotted-ключу). У хоста **нет**
  примитива для удаления ключа — поэтому откат к «ключа не было»
  реализован вручную: чтение сырого YAML, удаление dotted-ключа с
  чисткой опустевших родителей, запись обратно через
  `utils.atomic_yaml_write` (в обход `save_config()`, который не
  гарантирует байт-в-байт откат).

## 2. `/cost [дни]` — что означает каждое поле

Формируется в `report.py::render_cost`. Аргумент — число дней (по
умолчанию 7); строка вида `/cost 30.5` тоже валидна (дробные дни
допустимы).

- **«Нет данных в журнале за этот период»** — журнал пуст за это окно.
  Причины см. в разделе 6 «Диагностика».
- **Запросов** — число строк в таблице `requests` за окно (по одному
  ряду на каждый вызов API модели, не на сообщение пользователя —
  один ход агента может дать несколько запросов, например из-за
  ретраев или подзадач).
- **Токены — вход/выход/кэш-чтение/кэш-запись/рассуждения** — суммы по
  колонкам `input_tokens`, `output_tokens`, `cache_read_tokens`,
  `cache_write_tokens`, `reasoning_tokens` за окно. Это сырые данные из
  `usage`-словаря хука `post_api_request` — ничего не переоценивается.
- **Топ-5 моделей по токенам** — сумма input+output по модели,
  отсортировано по убыванию, показаны первые 5. Доллары рядом с моделью
  — **лучшая оценка**, не авторитетная цифра: считается через
  `agent.usage_pricing.estimate_usage_cost` по реальным токенам этой
  модели. Если модель неизвестна прайсеру или импорт не удался —
  честно «н/д», без выдумывания цифр.
- **Топ-5 сессий** — сумма input+output токенов по `session_id`,
  отсортировано **сначала по $** (авторитетная цена из
  `hermes_state.SessionDB.get_session(session_id).estimated_cost_usd`,
  а если её нет — `actual_cost_usd`), затем по токенам как тай-брейк.
  Если у сессии нет записи в `SessionDB` (например, старая/иная среда)
  — тоже «н/д», но сессия всё равно попадёт в список по токенам.
- **Hit-rate кэша** — `sum(cache_read_tokens) / sum(input_tokens)` за
  окно, по всем моделям сразу. `н/д`, если `input_tokens` за окно равен
  нулю (не с чем делить). Число рядом — «сбросов кэша»: сколько раз за
  окно модель/провайдер поменялись посреди одной сессии (см. раздел 4).
- **Предупреждение: сессия … hit-rate N% при M запросах** — печатается
  только для сессий с ≥5 запросов и hit-rate < 30%. Меньше 5 запросов —
  шум (первый запрос в сессии всегда даёт 0% кэш-чтения, это нормально,
  ещё нечего было закэшировать).
- **Ошибок (с повтором)** — строки из таблицы `errors` за окно и сколько
  из них были помечены `retryable=True` при записи хуком
  `api_request_error`.
- **Активные переключатели** — те тумблеры из Части 2, для которых в
  `config_backup.json` есть запись (значит, они включены прямо сейчас).

## 3. `/tokenguard status`

Построчно каждый ключ из `TOGGLE_KEYS` (`cheap_aux`, `cheap_delegation`,
`cache_1h`) со статусом «включён»/«выключен» — статус читается из факта
наличия записи в `config_backup.json` (`toggle in backups`), **не** из
самого `config.yaml` напрямую. Это значит: если кто-то вручную поправил
`config.yaml` в обход `/tokenguard enable`, статус здесь может разойтись
с реальным конфигом — в таком случае `config_backup.json` не будет
соответствовать текущим значениям ключей (см. правило аудита №5 в
разделе 4, «сверка backup-файла»).

Дальше — зарезервированные тумблеры (`cron_cascade`, `context_editing`)
с пометкой «появится позже»: они распознаются командой, но не сделают
ничего, кроме этого сообщения.

В конце — «Дешёвая модель: provider/model» (из
`token_guard.cheap_provider` / `token_guard.cheap_model`) либо просьба
её задать, если ещё не задана.

## 4. Правила аудита (`/tokenguard audit`)

`audit.py::run_audit()` — чистая функция, ничего не пишет в конфиг,
только возвращает список замечаний. Пять проверок, каждая обёрнута в
свой `try/except` — падение одной не останавливает остальные.

### Правило 1 — `prompt_caching.cache_ttl` = 5 минут при нескольких сессиях

Срабатывает (`warning`), если `cache_ttl` не задан или равен `"5m"` **и**
в журнале за 90 дней встречается больше одной уникальной `session_id`.

**Важно про ложные срабатывания:** в схеме журнала нет колонки
«платформа» (через что пришла сессия — telegram, discord, gateway…),
поэтому «есть ли гейтвей-сессии» — эвристика: «в журнале больше одной
уникальной сессии за 90 дней». Это не то же самое, что «реально
несколько параллельных клиентов», и может сработать даже у одного
пользователя, который просто открыл вторую сессию месяц назад. На
практике эвристика неплохо коррелирует с длительным/повторяющимся
использованием, но воспринимай warning как «стоит присмотреться», а не
как точный диагноз.

Подсказка: включить `cache_1h`.

### Правило 2 — модель сжатия контекста

- Если `auxiliary.compression.model` **не задан** → `info`: сжатие идёт
  основной (обычно дорогой) моделью. Подсказка: задать дешёвую модель и
  включить `cheap_aux`.
- Если задан — плагин пытается сравнить контекстное окно модели сжатия
  с основной моделью через `agent.model_metadata.get_model_context_length`.
  - Окно сжатия меньше основной → `warning`: «Окно модели сжатия меньше
    основной модели» — это и есть «ловушка компрессии»: модель с
    коротким окном молча теряет середину диалога при суммаризации, без
    явной ошибки. Подсказка: выбрать модель с окном не меньше основной.
  - Если сравнить не удалось (модель неизвестна метаданным, импорт не
    получился) → `info`: «Не удалось сравнить окна моделей
    автоматически», честная просьба проверить вручную. **Это не значит,
    что всё в порядке** — просто плагин не может утверждать ни то, ни
    другое.

### Правило 3 — модель делегирования

Если `delegation.model` не задан → `info`: сабагенты используют
основную модель и провайдера. Подсказка: задать дешёвую модель и
включить `cheap_delegation`.

### Правило 4 — неиспользуемые наборы инструментов

Требует **минимум 14 дней** истории в журнале (`ledger.history_span_days()`
считает от самой старой записи в `requests` до сейчас). Если истории
меньше 14 дней — правило тихо ничего не возвращает (не варнинг «мало
данных», а просто пропускает проверку). Отсюда и правило поведения:
если пользователь просит аудит на свежепоставленном плагине, честно
скажи, что часть проверок (в частности эта) заработает позже.

Когда истории достаточно: сопоставляет вызовы инструментов из журнала
(`tool_calls`) с наборами инструментов через
`model_tools.get_toolset_for_tool` (если хост-API недоступен — проверка
тихо пропускается) и для каждого **включённого**, но ни разу не
вызванного за это окно набора выдаёт `info`: «включён, но не
используется — кандидат на отключение, снижает размер схем в каждом
запросе». **Плагин не отключает ничего сам** — это только подсказка.

### Правило 5 — сверка backup-файла

Если в `config_backup.json` есть запись для тумблера, которого нет в
текущем реестре `TOGGLE_KEYS` (например, после отката версии плагина) —
`info` с просьбой проверить файл вручную. Штатная ситуация, встречается
редко, обычно после ручного вмешательства в `config_backup.json`.

## 5. Безопасный флоу каждого тумблера

Общая механика для всех трёх: `enable <toggle>` без `confirm` **всегда**
возвращает карточку риска и не трогает конфиг; `enable <toggle> confirm`
применяет; `disable <toggle>` откатывает без подтверждения. Перед первым
изменением ключей toggles.py делает снимок их текущих значений в
`config_backup.json` (или помечает `null`, если ключа не было) — этот
снимок и есть источник отката. Повторный `enable` уже включённого
тумблера НЕ трогает уже сохранённый снимок (чтобы не затереть
оригинал повторным включением).

### `cheap_aux`

Правит: `auxiliary.compression.model/provider`,
`auxiliary.title_generation.model/provider`.

Предусловие: должна быть задана `token_guard.cheap_model` +
`token_guard.cheap_provider` через `set-cheap-model` — иначе `enable`
(даже с `confirm`) вернёт просьбу сначала задать модель.

Карточка риска (дословно, как выводит плагин):
> cheap_aux — дешёвая модель для служебных задач (сжатие контекста,
> заголовки).
> Экономия: суммаризация/заголовки/поиск по сессиям — самые частые
> служебные вызовы.
> Риск: слабое резюме при сжатии теряет детали навсегда; модель с
> коротким окном молча выкидывает середину диалога.
> Страховка: ставьте длинноконтекстную «флеш»-модель; `/tokenguard
> audit` проверяет размер окна; откат одной командой — `/tokenguard
> disable cheap_aux`.

Типовые «длинноконтекстные дешёвые» модели для `cheap_aux` (ориентир
для помощи пользователю при выборе — конкретный список моделей и цен
меняется быстрее методички, сверяйся с актуальным прайсом провайдера):
компактные «flash/haiku/mini»-модели того же семейства, что и основная,
но с окном **не меньше**, чем у основной модели диалога. Если основная
модель — что-то с окном 200K+, дешёвая модель для сжатия должна быть
того же порядка, иначе именно она обрежет середину диалога при
суммаризации.

Важное расхождение с исходной спецификацией: в этой версии hermes
(0.18.0) блок `auxiliary.session_search.*` — мёртвый конфиг, движком
игнорируется (это видно из комментариев хоста). Поэтому `cheap_aux`
**не трогает** `session_search`, только `compression.*` и
`title_generation.*`. Если пользователь ожидает, что «поиск по сессиям»
тоже подешевеет — поясни, что в этой версии hermes он не настраивается
через данный ключ.

### `cheap_delegation`

Правит: `delegation.model/provider`. То же предусловие про
`set-cheap-model`.

Карточка риска:
> cheap_delegation — дешёвая модель для сабагентов (делегирование).
> Экономия: сабагенты (поиск, сбор информации) на дешёвой модели — до
> −50% на делегированиях.
> Риск: глубокие рассуждения в сабагенте могут просесть по качеству.
> Страховка: тяжёлые задачи не делегируйте или отключите тумблер; откат
> одной командой — `/tokenguard disable cheap_delegation`.

Обоснование цифры «до −50%» и что это не голословно: делегирование на
дешёвую модель для вспомогательных/поисковых сабагентов — практика,
проверенная в проде (телеметрия Claude Code показывает ~36% вызовов
сабагентов на Haiku при трёхуровневой маршрутизации моделей, что даёт
порядка −51% стоимости сессии). Здесь это ручной, явный тумблер, а не
автоматический роутинг — конкретную экономию в конкретном проекте
покажет `/cost` после нескольких дней с включённым тумблером.

Для глубоких многошаговых рассуждений внутри сабагента (не «найти
файл», а «спроектировать архитектуру») дешёвая модель — не всегда
хороший выбор; тумблер экономит на *частых* и *простых* делегированиях,
а не заменяет суждение о том, что стоит делегировать вообще.

### `cache_1h`

Правит: `prompt_caching.cache_ttl` → `"1h"`.

Карточка риска:
> cache_1h — кэш промптов на 1 час вместо 5 минут.
> Экономия: чтение кэша стоит ~10% обычной цены; час выгоден, если
> между сообщениями бывают паузы.
> Риск: запись в кэш на 1 час дороже (2× вместо 1.25×) — при очень
> редких сообщениях экономия может выйти в ноль.
> Страховка: `/cost` покажет hit-rate; откат одной командой —
> `/tokenguard disable cache_1h`.

Числа за карточкой (прайсинг Anthropic): чтение из кэша стоит ~10% от
цены обычного входного токена независимо от TTL; разница в цене записи
— 5-минутный кэш пишется по множителю ×1.25 от обычной цены, а
часовой — по ×2. То есть `cache_1h` окупается, только если пауза между
сообщениями в сессии обычно длиннее 5 минут, но короче часа: тогда
5-минутный кэш успевает протухнуть и переписаться заново (по полной
цене плюс запись), а часовой доживает до следующего сообщения и
отдаёт дешёвое чтение. При очень частых сообщениях (короче 5 минут)
разница нулевая — оба TTL живы. При очень редких (дольше часа) —
тоже нулевая, оба протухают, а часовой ещё и дороже писался.
Проверка на практике — смотреть `/cost` до и после: если hit-rate
кэша не подрос, а сбросов кэша (`cache_bust`, см. раздел 6) стало не
меньше — тумблер не помогает, можно откатить.

## 6. Диагностика

**Журнал пуст / `/cost` всегда «нет данных»:**
- Плагин не в `plugins.enabled` в `config.yaml` (или не установлен в
  `~/.hermes/plugins/token-guard` / `plugins/token-guard` внутри
  hermes-agent) — тогда `register(ctx)` вообще не вызывался, хуки не
  зарегистрированы, ничего не пишется.
- Плагин включён, но хуки почему-то не сработали ни разу — проверить,
  что версия hermes-agent действительно шлёт `post_api_request` (в
  этой сборке — v0.18.0, хуки подтверждены в коде `__init__.py`).
- Журнал есть, но за пределами запрошенного окна — попробовать
  `/cost 90` вместо `/cost` по умолчанию (7 дней).
- Файл `<HERMES_HOME>/token-guard/ledger.db` физически отсутствует или
  пуст — можно проверить наличие файла напрямую в файловой системе.

**`/tokenguard status` показывает «выключен», а пользователь уверен,
что включал:**
- Статус читается из `config_backup.json`, а не из `config.yaml`.
  Если кто-то вручную удалил `config_backup.json` или откатил его —
  статус «выключен», даже если ключи конфига всё ещё стоят от
  тумблера. В таком случае реальные значения `config.yaml` и
  `config_backup.json` разошлись — сюда же ведёт правило аудита №5.

**`enable <toggle> confirm` вернул ошибку "Не удалось применить
переключатель":**
- Скорее всего `set_config_value` упёрся в managed-конфигурацию (ключ
  заблокирован, например managed/NixOS-сборка) и вызвал `sys.exit(1)`
  внутри хоста — плагин ловит это (`SystemExit` — не `Exception`,
  специально отловлено отдельно) и вернул сообщение вместо падения
  всего процесса. В такой среде ключ придётся менять вне плагина, у
  администратора окружения.

**`disable <toggle>` говорит «не включён — нечего откатывать», хотя в
конфиге явно стоят чужие значения:**
- Значит для этого тумблера нет снимка в `config_backup.json` (либо
  никогда не включался через плагин, либо снимок был вручную стёрт).
  `disable` работает только от собственного снимка плагина, он не
  умеет «угадывать» исходное состояние конфига.

**Кэш «не работает» / hit-rate низкий:**
- Смотреть `/cost` → строку `Hit-rate кэша` и число «сбросов кэша».
  Частые `cache_bust` (событие пишется в `cache_guard.py` при смене
  `model`/`provider` внутри одной `session_id`) — самая частая причина:
  каждая смена модели посреди сессии обнуляет накопленный кэш, потому
  что кэш у большинства провайдеров привязан к точному id модели.
- Если `cache_bust` редкие, а hit-rate всё равно низкий — возможно,
  паузы между сообщениями превышают TTL кэша (обычно 5 минут) —
  кандидат на `cache_1h` (см. раздел 5), решение подтверждается тем же
  `/cost` после включения.

## 7. Чего плагин не делает (сознательно)

- **Не маршрутизирует запросы** между моделями автоматически — нет
  «умного роутинга» дешёвая/дорогая модель по сложности задачи. Такая
  функциональность (`smart_model_routing`) разрабатывается апстримом
  hermes (см. PR #53193/#58838/#59390) — token-guard намеренно её не
  дублирует, чтобы не конфликтовать с будущей штатной реализацией.
- **Не меняет компрессию контекста по содержанию** — не переписывает
  промпты сжатия, не решает, что оставить/выбросить при суммаризации.
  Тумблер `cheap_aux` только подставляет другую модель на существующий
  механизм сжатия хоста.
- **Не делает «ленивые» схемы инструментов** — токены на схемы
  инструментов (по факту ~21-26K токенов на запрос в стоковом hermes)
  плагин не сокращает и не подгружает по требованию; это тоже
  апстримная работа хоста, не token-guard.
- **Не отключает автоматически** неиспользуемые наборы инструментов —
  правило аудита №4 только подсказывает кандидата, финальное решение и
  действие — за пользователем.
- **Не хранит и не оценивает собственную таблицу цен** как источник
  истины — доллары по сессии всегда берутся из `hermes_state.SessionDB`
  (авторитетно), доллары по модели — best-effort оценка, которая может
  быть «н/д».
- **Не атомарно** — включение тумблера сначала пишет резервную копию,
  потом применяет ключи по очереди; если процесс прервётся между этими
  шагами, `disable` всё равно самовосстановится до консистентного
  состояния (снимок уже сохранён), но краткое окно частичного
  применения теоретически возможно.
