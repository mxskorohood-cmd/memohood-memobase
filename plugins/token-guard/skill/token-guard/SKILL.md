# token-guard

Скилл учит, как пользоваться плагином `token-guard`: журналом расхода
токенов, аудитом конфигурации и переключателями экономии (Часть 2).
Подробная методичка — в `references/guide.md`, читай её при любой
неочевидной ситуации (интерпретация полей, диагностика, типовые
дешёвые модели). Здесь — только репертуар команд и жёсткие правила.

## Когда применять этот скилл

Применяй, когда пользователь:

- спрашивает про расходы / токены / стоимость сессии или дня («сколько я
  потратил», «сколько токенов ушло», «покажи расход»);
- просит сэкономить или жалуется, что «дорого» / «сжигает токены»;
- просит аудит конфигурации, диагностику или «проверить настройки»;
- просит включить или выключить любой тумблер (`cheap_aux`,
  `cheap_delegation`, `cache_1h`);
- жалуется, что «агент поглупел», стал хуже суммировать или путается —
  это повод проверить активные тумблеры;
- спрашивает про кэш промптов / hit-rate / «кэш перестал работать».

## Репертуар команд

Все команды доступны и как слэш-команды в чате, и как CLI
(`hermes token-guard ...`). Ниже — слэш-форма; CLI-форма даёт то же
самое, просто без `/`.

### `/cost [дни]`

Отчёт о расходе за период (по умолчанию 7 дней).

```
/cost
/cost 30
```

Вывод содержит по порядку:
1. число запросов;
2. токены по корзинам: вход / выход / кэш-чтение / кэш-запись /
   рассуждения;
3. топ-5 моделей по токенам (с оценкой $, если цена известна —
   иначе «н/д»);
4. топ-5 сессий по $ (цена — от самого hermes, авторитетный источник);
5. hit-rate кэша и число сбросов кэша (`cache_bust`);
6. число ошибок API (и сколько из них с повтором);
7. список активных тумблеров.

Если данных за период нет — так и написано: «Нет данных в журнале за
этот период». Это не ошибка, а честный ответ на пустой журнал (см.
диагностику в guide.md).

### `/tokenguard status`

Показывает состояние каждого тумблера (включён/выключен) и
зарезервированные тумблеры (`cron_cascade`, `context_editing` — ещё не
реализованы), плюс заданную «дешёвую» модель, если она есть.

### `/tokenguard audit`

Прогоняет проверки конфигурации и журнала, возвращает список замечаний
(`[!]` — warning, `[i]` — info) с объяснением и подсказкой что делать.
**Ничего не меняет автоматически.** Полная расшифровка каждого правила —
в guide.md.

### `/tokenguard enable <toggle> [confirm]`

Двухшаговое включение тумблера:

```
/tokenguard enable cache_1h
```
→ возвращает карточку риска (экономия → риск → страховка) и просит
повторить с `confirm`.

```
/tokenguard enable cache_1h confirm
```
→ применяет изменение и подтверждает, что именно поменялось плюс команда
отката.

Тумблеры: `cheap_aux`, `cheap_delegation`, `cache_1h`. Для `cheap_aux`
и `cheap_delegation` сначала обязательно должна быть задана дешёвая
модель (`set-cheap-model`) — иначе `enable` вернёт просьбу сначала её
задать.

### `/tokenguard disable <toggle>`

Откатывает тумблер к состоянию до включения (восстанавливает прежние
значения ключей или удаляет ключ, если его не было). Подтверждение не
требуется.

### `/tokenguard set-cheap-model <provider> <model>`

Задаёт модель-провайдера, которую будут использовать `cheap_aux` и
`cheap_delegation` при включении.

```
/tokenguard set-cheap-model anthropic claude-haiku-4-5
```

## ЖЁСТКИЕ ПРАВИЛА ПОВЕДЕНИЯ

1. **Никогда не включай тумблеры Части 2 по своей инициативе.** Только
   по явной просьбе пользователя — и даже тогда сначала вызови
   `enable <toggle>` без `confirm`, получи карточку риска, **перешли её
   пользователю целиком** и дождись явного согласия. Только после этого
   вызывай `enable <toggle> confirm`. Не сокращай и не пересказывай
   карточку риска своими словами — показывай как есть: экономия/риск/
   страховка — это специально сформулированный текст.
2. **Выключение — без церемоний.** Если пользователь просит выключить
   тумблер, выполняй `disable <toggle>` сразу, без карточек и
   подтверждений.
3. **Перед `set-cheap-model` для `cheap_aux`** убедись, что модель
   длинноконтекстная (окно не меньше основной модели) — иначе сжатие
   контекста молча потеряет середину диалога. Если сомневаешься —
   предупреди пользователя об этом риске прямым текстом и посоветуй
   потом проверить `/tokenguard audit` (правило 2 сравнивает окна).
4. **Аудит имеет смысл только когда накоплена статистика.** Если
   `/tokenguard audit` возвращает мало данных или пользователь только
   что поставил плагин — честно скажи, что статистики за несколько дней
   ещё нет и часть правил (например, по неиспользуемым наборам
   инструментов) сработает только после ~14 дней журнала.
5. **На вопрос «сколько я потратил» — сразу `/cost`.** Не гадай по
   памяти и не оценивай на глаз — вызови команду и приведи её вывод.

## Сценарии-рецепты

- **Недельный обзор расходов** → `/cost 7` (или без аргумента), обрати
  внимание на топ-модели/топ-сессии и на активные тумблеры внизу отчёта.
- **«Агент поглупел»** → `/tokenguard status`, посмотри какие тумблеры
  включены; если это недавно включённый `cheap_aux`/`cheap_delegation` —
  предложи `/tokenguard disable <toggle>` как самый быстрый откат, потом
  при желании разобраться — `/tokenguard audit`.
- **«Кэш перестал работать» / просадка hit-rate** → `/cost` и посмотри
  строку `Hit-rate кэша` и `сбросов кэша`; частые `cache_bust` обычно
  значат, что модель/провайдер меняются посреди сессии (переключение
  моделей рушит кэш). См. guide.md, раздел про cache_guard.
